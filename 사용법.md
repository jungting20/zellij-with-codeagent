# Zellij Agent Runtime 사용법 및 설계 가이드

이 문서는 정호님과 Gemini CLI 에이전트가 나눈 대화를 바탕으로 정리한 `zellij-with-codeagent` 프로젝트의 활용 방법과 핵심 설계 원칙을 담고 있습니다.

## 1. 프로젝트 개요
AI 에이전트가 Zellij 터미널을 안전하고 체계적으로 제어할 수 있도록 돕는 런타임 환경(`agentd`)을 구축하는 프로젝트입니다.

### 핵심 역할 분담
*   **Planner (AI/Cognition)**: 무엇을 할지 결정 (예: "로그인 버그 수정해줘")
*   **Daemon (agentd/Orchestration)**: 터미널 운영 및 라이프사이클 관리 (Registry, Event Bus, Subscriptions)
*   **Zellij (Execution Runtime)**: 실제 프로세스가 실행되는 터미널 환경

---

## 2. 기본 실행 방법

### ① 데몬 서버 실행 (agentd)
API 요청을 받을 준비를 합니다.
```bash
go run ./cmd/agentd serve --socket /tmp/agentd.sock
```

### ② 직접 API 제어 (Curl)
*   **런타임 상태 조회**:
    ```bash
    curl --unix-socket /tmp/agentd.sock http://localhost/v1/runtime | jq .
    ```
*   **실행 계획 요청**:
    ```bash
    curl --unix-socket /tmp/agentd.sock http://localhost/v1/requests \
      -H 'Content-Type: application/json' \
      -d '{"type": "execution_plan", "payload": { ... }}'
    ```

## 3. 핵심 아키텍처 가이드

### 에이전트 간 데이터 전달 흐름
*   **권장 방식**: **중앙 관리자(Planner/Event Bus) 경유 방식**
    *   Pane끼리 직접 통신하지 않고, 각 Pane의 출력을 `agentd`가 감지하여 `Event`로 발행합니다.
    *   Planner가 이벤트를 보고 다음 Pane에게 명령(`SendInput`)을 내립니다.
    *   장점: 관측성 확보, 에이전트 간 느슨한 결합, 자동 복구 용이

### 데이터 획득 전략 (하이브리드 모델)
*   **구독 (Subscribe - Stream)**: 실시간 이벤트 감지용 (예: "테스트 실패" 즉시 감지)
*   **요청 (Snapshot/Dump - Pull)**: 정밀 상태 분석용 (예: "현재 코드 10라인 내용 보여줘")
*   **원칙**: 평소에는 구독으로 엿듣고, 필요할 때만 스냅샷을 요청하여 분석합니다.

---

## 4. 향후 워크플로우 시나리오
시스템 완성 시 다음과 같은 자율 워크플로우가 가능해집니다.

1.  **자율 버그 수정 (Self-Healing)**: 테스트 실패 감지 → 코드 분석 → 수정 → 재테스트 → 자동 종료
2.  **멀티 에이전트 협업**: 백엔드, 프론트엔드, 문서 에이전트가 각각의 팬에서 병렬로 작업하며 이벤트를 통해 협업
3.  **실시간 모니터링 및 복구**: 로그 감시 중 장애 패턴 감지 시 AI가 자동 복구 명령 하달

---

## 5. 주의사항
*   모든 Zellij 제어는 반드시 `agentd`의 API를 통해서만 이루어져야 합니다. (Planner가 직접 Zellij 명령어를 실행하면 안 됨)
*   Zellij 팬 ID는 일시적일 수 있으므로, 데몬이 관리하는 **Logical ID**를 기준으로 작업해야 합니다.
